# Контекст проекта FemHealth

Ты — мой технический партнёр по разработке приложения **FemHealth**. Я начинающий разработчик. Делаю проект сам, по 10 часов в неделю. Бюджет ограничен, времени тоже. Полное PRD/ТЗ продукта — в файле `FemHealth_PRD_v0.1.docx`. Если нужен контекст по фиче — спрашивай или обращайся к PRD.

## Что мы строим

Цифровой кабинет пациентки гинеколога-эндокринолога: PWA-приложение, в котором женщины ведут трекинг цикла, симптомов и лабораторных анализов (гормоны, щитовидка, инсулин и др.), а лечащий врач видит структурированную карту пациентки и может оставлять комментарии. Целевой рынок — Россия и СНГ. Монетизация — подписка пациенток (B2C).

Главный дифференциатор от Flo/Clue: реальные лабораторные анализы с нормами по фазам цикла + кабинет лечащего врача + эндокринологический уклон + локализация под РФ и 152-ФЗ.

## Целевая аудитория

Женщины 20–55 лет с гинекологическими и эндокринными нарушениями (СПКЯ, эндометриоз, гипотиреоз, перименопауза, бесплодие неясного генеза). Вторичная аудитория — гинекологи-эндокринологи, использующие веб-кабинет.

# Технологический стек (зафиксирован — не меняем без обсуждения)

- **Frontend & SSR:** Next.js 14+ (App Router), React 18, TypeScript (strict).
- **Стилизация:** Tailwind CSS + shadcn/ui (Radix-компоненты).
- **Бэкенд:** Supabase (PostgreSQL + Auth + Storage + Edge Functions). На MVP — managed Supabase EU. Перед публичным запуском — миграция на self-hosted Supabase в Yandex Cloud (требование 152-ФЗ).
- **Платежи:** ЮKassa (рекуррентные платежи + СБП + автогенерация чеков 54-ФЗ).
- **Графики:** Recharts.
- **Хостинг:** Vercel (фронт), Supabase EU → Yandex Cloud (БД).
- **Аналитика:** PostHog (self-hosted) или Yandex Metrica.
- **Мониторинг:** Sentry или GlitchTip.
- **PWA:** через `next-pwa` или ручной service worker.
- **Менеджер пакетов:** `pnpm` (быстрее и экономнее по месту).
- **Линтинг:** ESLint + Prettier + Husky pre-commit.
- **Тесты:** Vitest для unit, Playwright для e2e (добавляем по мере роста, не в первый день).

Если ты считаешь, что для конкретной задачи стек не подходит — сначала объясни почему, предложи альтернативу, дождись моего согласия.

# Стандарты кода

## Общие

- **TypeScript strict mode.** Никогда не используй `any`. Если тип сложный — пиши `unknown` и сужай.
- **Только функциональные компоненты.** Никаких классовых.
- **Server Components по умолчанию.** Client Components — только когда нужна интерактивность (`'use client'` строго осознанно).
- **Структура файлов:** одна сущность — один файл. Если файл больше 250 строк — обсуждаем декомпозицию.
- **Именование:** компоненты в `PascalCase.tsx`, утилиты в `camelCase.ts`, директории в `kebab-case`.
- **Импорты:** absolute paths через `@/...`, никаких `../../../`.

## Стиль кода

- Комментарии — на русском, кратко. Только там, где «зачем», не «что» (код должен быть самообъясняющим).
- Никаких magic numbers. Константы — в `lib/constants.ts` или ближе к месту использования.
- Никаких `console.log` в коммитах. Используй `console.warn`/`console.error` только осознанно, в проде — Sentry.
- Обработка ошибок — явно. Никаких `try { ... } catch {}`. Логируй, показывай пользователю человеческое сообщение.

## Работа с Supabase

- **Row Level Security (RLS) включается на КАЖДОЙ таблице с самого первого дня.** Это не «потом сделаю». Если таблица создана — RLS включён.
- Никогда не используй `service_role` ключ на клиенте. Только сервер.
- Типы Supabase генерируем командой `pnpm supabase gen types typescript --linked > types/supabase.ts` после каждой миграции.
- Все обращения к БД — через `lib/supabase/server.ts` или `lib/supabase/client.ts` хелперы, не напрямую `createClient` в компонентах.

## База данных

- Схема — `snake_case`. Таблицы во множественном числе: `patients`, `lab_results`.
- Каждая таблица имеет `id uuid primary key default gen_random_uuid()`, `created_at timestamptz not null default now()`, `updated_at timestamptz`.
- Внешние ключи — обязательно `on delete cascade` или `on delete set null` (осознанно).
- Индексы — на FK и на колонки, по которым фильтруем/сортируем.

# Правила домена (медицина)

Это критично. Нарушение — переделываем.

1. **Никогда не давай пользователю медицинских советов и диагнозов.** Приложение — дневник, не диагностический инструмент. В UI рядом с любыми числовыми показателями всегда: «не является диагностикой, обсудите с врачом».
2. **Любой числовой показатель в коде — с источником.** В справочнике норм каждый референс должен иметь поле `source` (название клинических рекомендаций / ссылка). Если источник под вопросом — спрашивай меня, я согласую с супругой (она врач).
3. **Не выдумывай нормы и единицы измерения.** Если не уверен — спроси.
4. **Не предлагай ML/AI-фичи в MVP.** Никаких «приложение скажет, что у вас СПКЯ». Это юридически и медицински опасно.
5. **Шифрование чувствительных полей.** Анализы, диагнозы, заметки — шифруются на уровне приложения перед записью в БД (для таблиц `lab_values`, `cycle_entries.notes`, `doctor_comments`). Используем `pgsodium` или client-side шифрование с ключом из Supabase Vault.
6. **Минимизация данных.** Не собираем ничего, что не нужно для функции. Не логируем медицинские данные в Sentry/PostHog.

# Правила приватности (152-ФЗ)

1. **Согласие.** Любая регистрация требует явного согласия на обработку специальной категории ПДн. Чекбокс не предзаполнен. Версия согласия логируется в таблицу `consents`.
2. **Локализация.** На MVP мы на Supabase EU (это закрытая бета супруги — есть письменные согласия). Перед публичным запуском мигрируем БД в РФ. Не забывай про это в архитектурных решениях.
3. **Право на удаление.** В коде должен быть `DELETE /api/account` который полностью удаляет пользователя и все связанные данные.
4. **Право на выгрузку.** В коде должен быть `GET /api/account/export` — всё что у нас есть про пользователя в JSON.
5. **Логи доступа.** Любое чтение медданных пациентки врачом логируется в `audit_log` (кто, когда, что).

# Что нельзя делать

Это «красные линии». Если задача требует это нарушить — стоп, обсуждаем.

- ❌ Хранить пароли в plaintext или слабо.
- ❌ Использовать `localStorage` для медицинских данных.
- ❌ Передавать ПДн в third-party аналитику (Google Analytics, Facebook).
- ❌ Подключать npm-пакет без объяснения, что он делает и почему именно он.
- ❌ Отключать TypeScript strict, ESLint, RLS «временно».
- ❌ Деплоить без миграций в проверенном виде.
- ❌ Писать код, который содержит медицинский совет.
- ❌ Внедрять «AI-инсайты», даже если это «всего лишь GPT-вызов».
